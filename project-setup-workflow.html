Phase 0 — Auto-Trigger & Pre-flight

AUTOMATED
Run by Claude • triggered by ClickUp status change
  1. Sales task moved to “Won” — ClickUp fires webhook to Make.com
    The person who moves the task to Won triggers the entire automation chain. Make.com receives the webhook with the task ID and the user ID of who changed the status, then calls Claude’s remote trigger.
  2. Pre-flight validation — check all required fields
    Before creating anything, every required field is checked. Contact Name, Email, Phone, Deal Value, Job Address, Plans, Proposal, and Buildsoft (BUILD only). If any are missing → setup STOPS immediately.
  3. Missing fields? Assigned comment sent to salesperson
    If fields are missing, an ASSIGNED COMMENT is posted on the sales task — it goes into the salesperson’s Assigned Comments tray (the red badge they can’t ignore). The comment lists exactly which fields need to be filled. Once fixed, toggle the status back to Won to re-trigger.
  4. All fields present? Proceed to Phase 1
    Pre-flight passed. The template provisioning begins — folder is created from the SBI PROJECT TEMPLATE and the system polls every 10 seconds (up to 3 minutes) until all 10 lists and the Project Doc are ready.

Phase 1 — Sales Handover

AUTOMATED
Run by Claude • ~2 min
  1. Project folder created from template (10 ClickUp lists)
    The SBI PROJECT TEMPLATE is applied to create a new folder in the SBI PROJECTS space. This gives us:
    10 standard lists (01 Sales File Info, 02 Design, 03 Admin, 04 Claims, 05 PM, 06 Schedule, 07 Pre-Manufacture, 08 Procurement, 09 Manufacturing, 10 Site Works) plus all template tasks, statuses, and custom fields.

    If a folder with the same job number already exists, it is reused — no duplicates created.
  2. Project Doc renamed to: ✨️11 “{jobNumber} {projectName}”
    The template doc is renamed from the generic template name to include the project number and name. Example: ✨️11 “260306 Natural Foot Podiatry Erina Fair”. This makes it easy to identify in ClickUp’s doc sidebar.
  3. Folder renamed: YYMMDD BUILD/DESIGN Project Name
    Examples:
    • 260306 BUILD Natural Foot Podiatry Erina Fair
    • 260317 DESIGN Tracey Fletcher Wamberal

    Project type is detected from the task name first (DESIGN/BUILD keyword), then falls back to the Job Type custom field. The Schedule list (06) is also renamed to include the full folder name.
  4. Template tasks customised with job name
    Placeholder names in template tasks are replaced with the actual job name:

    List 02 (Design):
    • {Job Name} - Construction Plans (All Comments)
    • {Job Name} - Drawing - Revision 0
    • {Job Name} - Drawing Approved for Construction

    List 08 (Procurement):
    • 300 {Job Name} BUILD LABOUR
    • 301 {Job Name} PM/OFFICE/SALES
    • 302 {Job Name} CAD/Design LABOUR
    • 303_304 {Job Name} JOINERY/FACTORY LABOUR
  5. Project Info Doc populated (client details, plans)
    The ClickUp Doc (List 11, page 1) is updated with:
    • Client contact details — name, email, phone
    • Job address
    • Construction plans reference table (links from sales task)
    • Joinery plans reference table (if joinery is in scope)

    This becomes the site reference document — the PM and site crew refer to it for key project info.
  6. Sales task multi-listed into List 01, prefixed “SOLD”
    Three actions in sequence:

    1. The original sales task is added to List 01 using ClickUp’s multi-list feature — same task in both the Deals list and the project folder.
    2. Task name prefixed with “SOLD” (e.g. SOLD TS260317 Tracey Fletcher...) — visible in both locations.
    3. The “ADD WON sales Task” placeholder in List 01 is renamed and marked done.

    After this, List 01 has two “SOLD” tasks: the completed placeholder and the multi-listed sales task with all the real data.
  7. Progress claims created in List 04
    Invoice milestone tasks created in List 04, all assigned to Gail.

    Claim%Stage
    Progress Claim 140%Deposit
    Progress Claim 230%Start on Site
    Progress Claim 320%Practical Completion
    Progress Claim 410%Final
    Claim%Stage
    Design Deposit Invoice60%Deposit
    Design Final Invoice40%Final
    Task names include the dollar amount, e.g.:
    Progress Claim 1 - Deposit 40% ($31,600 incl GST)
  8. Schedule set in List 06 (sequential phase dates)
    BUILD — 5 high-level tasks get sequential dates from date sold:
    TaskStartEnd
    JOB SOLD—Date Sold
    08 ProcurementDate Sold+ 1 week
    07 Pre-Manufacture+ 1 week+ 2 weeks
    09 Manufacturing+ 1 week+ 2 weeks
    10 SITE WORKS+ 2 weeks+ 3 weeks
    These are placeholder durations — the PM adjusts as the real timeline becomes clear.     DESIGN — Simplified schedule:
    • JOB SOLD — due date = date sold
    • Design Start — due date = date sold
    • Design Finish — due date = 3 weeks later
    • Build-phase tasks have dates cleared

    Important: Only the 5 high-level tasks are touched. Subtasks under 10.1 SITE WORKS are PM-owned and never modified.
  9. Site works tasks created from proposal scope BUILD ONLY
    The signed proposal and cost budget are read to extract the scope of works. This is broken into 5–10 sequential install tasks, created as subtasks of the 10.1 SITE WORKS task in List 06.

    Typical sequence:

    2. Site measure & mark out
    3. Protect existing equipment & floor coverings
    4. Material prep / cut to size
    5. Main install (scope-specific)
    6. Inspect alignment / quality
    7. Final clean & rubbish removal
    8. Site demobilisation & client walkthrough

    Task names reference actual materials and quantities from the proposal. A scope summary is also posted as a comment on the Completions_Punchlist task in List 10.
  10. Handover task assigned — Chadd, Gemma, Gail, Michael
    BUILD projects: The “Handover between SALES and PM” task in List 05 is updated with a due date (within 2 days) and assigned to the full team.

    The handover meeting covers:
    • Review Project Info Doc — client details, special requirements
    • Review sales documents — what was promised
    • Review construction plans and joinery plans
    • Discuss timeline, budget, risks
    • Assign next actions

    DESIGN projects (salesperson = designer): Handover is skipped — same person doing the work.
  11. Sales task updated: “Project Folder Ready” ticked
    • “SBI Project Folder Ready” checkbox ticked on the sales task
    • Comment posted with a link to the new project folder
    • Handover summary listing all data transferred, missing fields, and next actions

Phase 2 — Cost Budget

BUILD ONLY AUTOMATED
Run by Claude • ~1 min
  1. Buildsoft file located in 1.2 COST SHEETS/
    The Buildsoft Excel file is found on the shared drive at:
    G:\Shared drives\SBI JOB FILES_Share\{project}\1.2 COST SHEETS\

    The file contains cost code categories (300 Labour, 310 Materials, 320 Contractors, etc.) with quantities, rates, and totals from the estimating process.
  2. Pre-flight check: verify pricing data exists
    Why this matters: Sometimes a Buildsoft file has the cost code structure set up but no actual numbers entered yet. Running the budget generator on an empty file wastes time and produces $0 results.

    The check looks for at least one cost code with a Qty, Rate, and Total > 0. If the file is empty, you’re given three options:
    • Price it — go back and enter the numbers, then re-run
    • Mark N/A — skip budget + procurement for this project
    • Generate stub — reverse-engineer from deal value (estimates only)
  3. SBI Cost Budget tab generated in Excel
    A new “SBI Cost Budget” sheet is added to a working copy of the Buildsoft (never the original). It contains two tables:

    Cost Sheet Summary (top) — one row per cost code with Budget, Rate/hr, Hours Allowed, Man Days
    Cost Sheet Detail (bottom) — every line item with Qty × Rate = Budget

    Footer shows: Total Cost, Markup, Overhead Recovery (66%), Budgeted Profit (33%), Sell ex-GST, GST, Sell incl-GST.
  4. Formatted + exported to styled PDF with donut charts
    The Excel tab is styled with professional formatting and exported to a single-page PDF:

    • Dark navy headers, blue categories, grey subtotals, peach sell prices
    • Two donut charts side-by-side:
     — Cost Breakdown — one slice per cost code category
     — Sales Breakdown — Cost vs Overhead vs Profit vs GST

    Saved as SBI Cost Budget - {project name}.pdf
  5. Totals verified against deal value
    The Sell incl-GST from the cost sheet is compared against the Deal Value on the sales task. If they don’t match, it’s flagged — the Buildsoft may have been adjusted after the proposal was sent.

Phase 3 — Procurement Setup

AUTOMATED
Run by Claude • ~1 min
  1. Cost codes read from SBI Cost Budget tab
    Each cost code category and its budget amount is extracted from the Cost Sheet Summary table generated in Phase 2. Zero-budget categories are skipped.
  2. One task per cost code created in List 08
    Each cost code becomes a ClickUp task in List 08 (Procurement/Budget Track).

    Task name format: {code} {Job Name} {CATEGORY}
    Example: 310 Sunshine Kebabs BUILD Materials

    If re-running, existing tasks are matched by cost code number and updated (no duplicates).
  3. Budget amounts set on each task
    The Budget Allowance SBI custom field is set to the budget amount (ex GST) from the cost sheet. This is the figure the PM tracks against when placing orders.

    The dashboard later shows Budget vs Actual bar charts using this field alongside the Actual PO Spend field.
  4. Procurement category + workflow set
    Each task is categorised for filtering and reporting:
    CodeCategoryLabel
    300–304Labour - SBI InternalLABOUR
    310Build Works SupplierBUILD supplier
    315, 317Factory MaterialsJOINERY PROCUREMENT
    320–330ContractorsCONTRACTORS
    Workflow is set to “Not Started”. The PM progresses it through: Quoting → Quoted → Approved → Ordered → Received → Paid.

Phase 4 — Dashboard

AUTOMATED
Run by Claude • ~2 min
  1. All project data fetched from ClickUp
    Data is pulled from every project list:
    • Sales info — client details, deal value
    • Claims & variations (List 04) — amounts, status
    • Schedule (List 06) — phase dates, site works progress
    • Budget vs actual (List 08) — budget, PO spend, balance
    • Procurement status (List 08) — Not Started / Ordered / Received
    • Compliance (List 03) — admin task checklist
    • Site works (subtasks of 10.1 in List 06) — task status & dates
  2. HTML dashboard generated from template
    A self-contained HTML dashboard is generated with these tabs:

    • Overview — health badge, phase indicator, key metrics
    • Sales — client info, claims table, variations
    • Financials — budget vs actual bar charts per cost code
    • Procurement — workflow summary cards + task table
    • Schedule — Gantt chart of phases and milestones
    • Compliance — admin task checklist
    • Actions — next actions and blockers for Tuesday meeting
  3. Saved to shared drive + deployed to GitHub Pages
    The dashboard is saved to three locations:
    1. Local working copy
    2. Shared drive — @Claude Dashboards/{jobNumber}-{slug}.html
    3. GitHub Pages — michael-sbi.github.io/sbi-dashboards/{slug}/

    GitHub Pages deployment is automatic via git push. The dashboard is live within 1–2 minutes.
  4. Link posted to Projects Meeting task
    For new projects, a comment with the dashboard URL is posted on the current Projects Meeting task (reviewed every Tuesday).

    Format: 📊 {Project Name} — Project Dashboard: https://michael-sbi.github.io/sbi-dashboards/{slug}/

    The comment tags Michael (48602901), Gail (48603029), and Leah (100992399) so they all get ClickUp notifications (in-app + email). Each person’s manual steps are listed next to their name.
